## Context

Vault-файлы генерируются из `.hbk` HTML-справки 1С. После конвертации HTML → Markdown информация о типах существует как проза: `Тип: [Строка](link).` в теле параметров и возвращаемых значений. Ссылки на типы уже разрешены в vault-пути на этапе `rewrite_links`. Нужен постобработчик который читает готовые `.md`, извлекает типовую информацию и вставляет компактную сигнатурную строку.

Целевой формат (вариант Б — inline backticks + MD-ссылки):
```
`НайтиФайлы`(`<Путь>`: [`Строка`](lang__def_String.md), `<Маска>`: [`Строка`](lang__def_String.md), `<ИскатьВПодкаталогах>`: [`Булево`](lang__Булево_(Boolean).md)) → [`Массив`](Массив_(Array).md)
```

Для свойств:
```
`ЗащищенноеСоединение`: [`ЗащищенноеСоединениеOpenSSL`](link) | [`Неопределено`](link)
```

## Goals / Non-Goals

**Goals:**
- Постобработчик парсит готовые `.md` из vault — не трогает HTML-парсинг.
- Вставка сигнатуры после H1 и breadcrumb, перед основным телом страницы.
- Поддержка: методы (включая многовариантные), свойства, конструкторы.
- Составные типы через `|`, ссылки сохраняются.
- Новый этап `STAGE_INJECT_SIGNATURES` в pipeline, статистика в Stats.

**Non-Goals:**
- Не менять HTML-парсинг или этапы до `inject_breadcrumbs`.
- Не трогать `_meta.json` — структура метаданных не изменяется.
- Не генерировать сигнатуры для index-файлов (`_index*.md`).
- Не добавлять YAML frontmatter.
- Не парсировать `.hbk` заново — только готовый Markdown.

## Decisions

### 1) Парсить Markdown, а не HTML

**Decision:** Постобработчик читает готовые `.md` vault-файлы, а не возвращается к HTML.

**Why:** HTML недоступен после конвертации (временная директория удалена). Markdown уже содержит все нужные данные с разрешёнными ссылками.

**Alternative considered:** Извлекать типы на этапе `convert_one` из BeautifulSoup.
**Rejected:** Усложняет горячий путь конвертации; типы нужны только для постобработки; BeautifulSoup в convert_one уже удалён к моменту записи файла.

### 2) Определение типа страницы по структуре Markdown

**Decision:** Тип страницы (метод / свойство / конструктор) определять по наличию структурных маркеров в тексте:
- `Синтаксис:` → метод или конструктор
- `Вариант синтаксиса:` → многовариантный метод
- `Возвращаемое значение:` → метод с return-типом
- Раздел `Использование:` (Только чтение / Чтение и запись) → свойство
- Отсутствие `Синтаксис:` и наличие `Тип:` в начале описания → свойство

**Alternative considered:** Определять по `source_path` из `pages_meta`.
**Rejected:** Постобработчик работает с vault-файлами независимо; привязка к `pages_meta` создаёт зависимость.

### 3) Regex-парсинг "Тип:" строк

**Decision:** Использовать regex для извлечения типов из строк вида `Тип: [Name](link), [Name2](link2).`

Паттерн: `Тип:\s*(.+?)\.?\s*$` с последующим разбором MD-ссылок `\[([^\]]+)\]\(([^)]+)\)` или plain-текстовых токенов.

**Why:** Структура "Тип:" строк в vault стабильна — это результат прямой конвертации HTML секций. Regex достаточен и предсказуем.

**Alternative considered:** BeautifulSoup / HTML-парсинг на этапе convert.
**Rejected:** см. Decision 1.

### 4) Место вставки сигнатуры

**Decision:** Вставлять сигнатуру **после** H1 и breadcrumb, **перед** первым контентным блоком (объект/метод имя, версия, синтаксис).

Структура результата:
```
# Заголовок

**↑** [Главная](...) › ...

`Метод`(`<Param>`: [`Тип`](link)) → [`ReturnType`](link)

ОбъектИмя (ObjectName)
...
```

**Why:** Сигнатура — самый важный элемент для агентов. Размещение сразу после навигации гарантирует попадание в первый RAG-чанк. Не нарушает H1-first правило inject_breadcrumbs.

### 5) Многовариантные методы

**Decision:** Каждый синтаксический вариант — отдельная строка сигнатуры. Варианты разделяются пустой строкой.

```
`Записать`(`<ИмяВходногоФайла>`: [`Строка`](link), `<АдресРесурса>`: [`Строка`](link))
`Записать`(`<АдресРесурса>`: [`Строка`](link), `<ВходнойПоток>`: [`Поток`](link) | [`ПотокВПамяти`](link) | [`ФайловыйПоток`](link))
```

**Why:** Полная информация без потери вариантов. Агент видит все overloads.

### 6) Идемпотентность

**Decision:** Перед вставкой проверять наличие существующей сигнатуры (маркер-строка начинается с `` ` ``  после breadcrumb). Если есть — заменять, не дублировать.

**Why:** Позволяет повторный запуск конвертации без накопления дублей.

## Risks / Trade-offs

- **[Risk] Regex не распознаёт нестандартный формат "Тип:"** → Mitigation: если парсинг не даёт результата — страница пропускается без сигнатуры, без ошибки; счётчик `signatures_skipped` в Stats.
- **[Risk] Свойства без явного "Тип:" блока** → Mitigation: только секция с `Тип:` в "Описание:" используется; если отсутствует — пропуск.
- **[Risk] Регресс inject_breadcrumbs** → Mitigation: `STAGE_INJECT_SIGNATURES` выполняется после `inject_breadcrumbs`; breadcrumb-логика не затрагивается.
- **[Trade-off] Постобработка = второй проход по всем файлам** → Приемлемо: файлов ~5-10k, операция I/O-bound, добавляет ~секунды к общему времени.
