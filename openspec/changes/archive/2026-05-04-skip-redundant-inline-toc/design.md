## Context

Сегодня `render_inline_toc` в `convert.py` строит markdown TOC только из дерева `TreeNode` и всегда дописывает блок в конец файла (если у узла есть дети и сегмент не в `SKIP_SEGMENT_INDEX` для самой страницы). Часть подразделов получает пустой `target` (нет `index_name_map` для сегментов из `SKIP_SEGMENT_INDEX`), что даёт `]()`. Требования к избыточности и «прочности» согласованы с Павлом в обсуждении перед propose.

## Goals / Non-Goals

**Goals:**

- Вставлять блок TOC **только если** есть хотя бы одна **валидная** ссылка TOC (непустой путь, файл существует в `out_dir`), для которой в теле страницы **нет** литерала `](path)`.
- Не выводить в TOC пункты с пустым target; не использовать их для решения «показать / скрыть» (вакуумно «покрыты»).
- Цели на **несуществующие** `.md` в `out_dir` не выводить в TOC и не считать «незакрытыми» телом (вакуумно «покрыты»).
- Сканирование тела: без строки крошек (`**↑**` …) и без существующего блока `<!-- toc:start -->`…`<!-- toc:end -->`; без нормализации путей (`./`, якоря, encodings).
- `inject_inline_toc_into_content("")` или эквивалент: убрать блок TOC с диска, если решение — не показывать.

**Non-Goals:**

- Отдельный CLI/стадия «пересобрать только TOC».
- Зависимость логики TOC от наличия или порядка вставки крошек (крошки исключаются из текста для сканирования; порядок этапов не должен менять итоговое решение — при реализации читать текущий файл и вырезать известные префиксы).
- Нормализация markdown-ссылок или поддержка `obsidian://` в критерии совпадения.

## Decisions

1. **Единый источник «какие ссылки в TOC»** — тот же расчёт, что сейчас даёт пары `(label, target)` для подразделов и страниц, плюс фильтр: `target` непустой и `(out_dir / target).is_file()`.

2. **Избыточность** — для множества валидных целей `T` из п.1: TOC избыточен, если `∀ p ∈ T` в тексте для сканирования есть подстрока `](p)` (буквальное совпадение строки пути как в TOC).

3. **Порядок этапов** — при одном проходе: при инжекции TOC читать файл, вычислить `scan_text`, вызвать рендер TOC с `out_dir` / `scan_text`; не полагаться на крошки внутри `scan_text`. Если крошки вставляются тем же пайплайном до или после TOC, `scan_text` должен использовать актуальное содержимое с вырезанной строкой крошек (и без старого TOC), чтобы критерий не зависел от порядка вставки.

4. **Пустой результат** — если после фильтра валидных целей множество `T` пусто, не вставлять блок; если блок был — удалить (идемпотентная запись).

**Альтернативы отвергнуты:** построчное удаление только «Подразделов» (ранний гибрид) — пользователь выбрал all-or-nothing для всего блока.

## Risks / Trade-offs

- **[Risk]** Тело содержит тот же путь в «нестандартной» форме (`./x.md`, другой регистр на чувствительной ФС) — не засчитается, TOC останется. **→ Mitigation:** зафиксировано в спеке как осознанное ограничение (литерал как в TOC).

- **[Risk]** Две разные страницы с одинаковым отображаемым текстом но разными путями — критерий только по `path`, без ложных срабатываний по тексту ссылки.

- **[Risk]** Гонка порядка breadcrumb vs TOC — **→ Mitigation:** явное вырезание крошек и TOC из буфера перед сканированием (см. решение 3).

## Migration Plan

Полная переконвертация `convert.py --clean` (или принятый в проекте сценарий) пересоздаёт vault; отдельной миграции не требуется.
